La solución de MitrufelyWeb está diseñada bajo un modelo moderno distribuido. Distribuimos cada componente en servicios especializados respetando el principio de separación de responsabilidades para optimizar el consumo de recursos en la nube y asegurar una escalabilidad simplificada.

1. Topología de Despliegue Distribuida

El sistema se distribuye sobre cuatro servicios independientes en proveedores especializados según su rol de arquitectura:

Componente Tecnología Plataforma de Despliegue Función
Frontend (SPA) React 19 + Vite (build estático) Vercel Servidor de archivos estáticos optimizado con CDN global.
Backend (API REST) FastAPI + Uvicorn Render (Web Service) Servidor de aplicación ASGI con escalado horizontal.
Workers asíncronos Celery (Worker + Beat) Render (Background Worker) Procesamiento asíncrono (PDF, Excel, analítica, expiraciones).
Caché / Cola Redis Render (Redis) Broker de Celery + persistencia de carritos y blocklist.
Base de Datos PostgreSQL NeonDB (Serverless) Persistencia transaccional ACID con triggers y vistas.
Imágenes — Cloudinary CDN de distribución de imágenes (fotos de productos).
Comunicación entre Componentes

El frontend consume la API mediante su URL pública (VITE_API_BASE_URL). El backend se conecta a NeonDB, Redis y Cloudinary por medio de cadenas seguras inyectadas en tiempo de ejecución. Esto permite que cada componente escale, se monitoree y se actualice de forma totalmente independiente.

2. Despliegue del Frontend en Vercel (Edge Network)

El frontend es empaquetado como una SPA estática optimizada para renderizarse eficientemente en la red de borde global de Vercel:

  • Build de producción: El comando npm run build ejecuta el typecheck de TypeScript (tsc -b) y luego empaqueta con Vite aplicando minificación y code-splitting en la carpeta dist/.
  • Publicación continua: La integración de Vercel con GitHub detecta los pushes en la rama master para compilar y desplegar automáticamente la última versión en un dominio seguro *.vercel.app.
  • Edge Network: Los artefactos estáticos se replican en los puntos de presencia (PoP) de Vercel alrededor del mundo. Esto minimiza el tiempo de carga inicial y la latencia para el usuario final.
  • Variables de entorno: La URL del backend se inyecta en el build mediante VITE_API_BASE_URL apuntando a la API desplegada en Render.

3. Despliegue del Backend en Render (Contenedores Docker)

El backend de FastAPI se ejecuta en Render a través de un contenedor Docker multi-stage para garantizar la paridad absoluta entre desarrollo y producción.

i. Estrategia de Imagen Multi-Stage

El Dockerfile define tres etapas para reducir el tamaño de la imagen final y reforzar la seguridad:

  1. Builder: Descarga e instala las librerías en un entorno aislado sin compiladores innecesarios.
  2. Development: Monta dependencias de desarrollo y permite la ejecución en caliente con --reload para desarrollo ágil en docker-compose.
  3. Production: Basado en una imagen mínima de python:3.11-slim, copia solo las librerías construidas, corre bajo el usuario no privilegiado appuser y define un HEALTHCHECK propio.

ii. Orquestación en Producción (render.yaml)

La infraestructura se autodefine como código a través de render.yaml:

  • Web Service (FastAPI): Ejecuta Uvicorn con dos workers de aplicación. Incluye un Health Check en /api/v1/health que Render monitorea constantemente para validar la disponibilidad del contenedor antes de cambiar el tráfico.
  • Background Worker (Celery): Inicia el worker y el beat scheduler de forma separada del hilo de la API, encargándose de la expiración automática de lotes/cupones y reportes pesados.
  • Redis (broker/caché): Configurado bajo una red privada sin exposición a internet (ipAllowList vacía) y con política allkeys-lru.

4. Gestión de Configuración y Secretos

Todas las variables sensibles se gestionan fuera del código fuente e inyectadas por los proveedores de hosting:

Variable de Entorno Propósito Origen / Almacenamiento
DATABASE_URL Conexión transaccional a NeonDB Inyectada en Render como secreto protegido.
SECRET_KEY Semilla de firma de tokens JWT Generada aleatoriamente en el panel de Render.
REDIS_URL Conexión al broker interno Redis Provista de forma automática por la red de Render.
APP_ENV / DEBUG Define entorno (production / false) Inyectado manualmente en los contenedores.
VITE_API_BASE_URL URL pública de la API de FastAPI Definida en Vercel para el empaquetado.
CLOUDINARY_* Credenciales del CDN de medios Inyectadas en Render para subir fotos.
Patrón Fail-Fast con Pydantic Settings

El backend lee estas variables al arrancar usando Pydantic Settings estructurado con el decorador @lru_cache para un acceso ágil (Singleton). Si alguna variable requerida falta o tiene un formato incorrecto, la aplicación arroja un fallo inmediato durante el arranque impidiendo que el contenedor inicie de forma corrupta.

5. Microservicio de Entregas (Delivery Service)

El simulador asíncrono se mantiene como una pieza independiente del backend transaccional:

  • Se levanta como un servicio FastAPI en el puerto 8001, exponiendo los endpoints de gestión de entregas.
  • Una vez simulado el tránsito, invoca al webhook de finalización del backend firmando la petición con un token criptográfico (x-delivery-token).
  • Aplica una estrategia de reintentos asíncronos bajo la fórmula de **Backoff Exponencial** ante fallos de conexión temporales.
  • Este aislamiento permite migrar el simulador a una API de un proveedor real en el futuro sin realizar modificaciones estructurales en el backend.

6. Estrategia CI/CD y Pipeline de Liberación

Se adopta un flujo GitOps de entrega continua basado en integraciones con GitHub:

[Push a master]
       │
       ├──► (Vercel): Compilación estática y distribución global en CDN Edge.
       │
       └──► (Render): Construcción de imagen Docker y despliegue del Web Service.
               │
          [Health Check (/api/v1/health) responde OK]
               │
          [Enrutamiento del tráfico de usuarios] (Zero-Downtime)
  • Zero-Downtime Deployment: Render mantiene el contenedor antiguo activo sirviendo tráfico hasta que el nuevo pasa el Health Check en /api/v1/health.
  • Estrategia de Rollback: Ante fallos inesperados detectados post-despliegue, se puede revertir a un estado estable anterior con un solo clic desde la consola de Vercel/Render.

7. Consideraciones de los Planes Free y Escalabilidad

Para mitigar las restricciones impuestas por los planes gratuitos, se adaptaron las siguientes soluciones arquitectónicas:

  • Render Idle Time: El Web Service gratuito de Render entra en suspensión tras 15 minutos sin tráfico. La arquitectura asíncrona mediante Celery asegura que la primera petición entrante tras la reactivación se encole correctamente sin generar pérdidas de transacciones.
  • NeonDB Auto-suspend: La base de datos Postgres serverless escala a cero en periodos de inactividad. El pool de conexiones asíncronas de SQLAlchemy (asyncpg) gestiona las reconexiones de forma transparente sin crashear la aplicación.
  • Vercel Bandwidth: El límite del plan Hobby es adecuado gracias al cacheo de archivos en la CDN de Vercel, lo cual reduce significativamente la cantidad de peticiones que llegan al servidor de origen.
  • Escalado Horizontal Futuro: El desacoplamiento absoluto de las capas (frontend estático CDN, API sin estado, colas Redis independientes y base de datos serverless) permite migrar a planes de pago con escalado dinámico simplemente ajustando las configuraciones de los proveedores de nube.
📖 Glosario

📖 Glosario de Despliegue

Cloud-Native
Diseño y construcción de aplicaciones optimizadas para ejecutarse en entornos de nube distribuida y desacoplada.
SPA (Single Page Application)
Aplicación web que carga una única página HTML y actualiza dinámicamente el contenido mediante JavaScript (React) a medida que el usuario interactúa.
CDN (Content Delivery Network)
Red distribuida de servidores que sirven contenido estático (archivos JS, CSS, imágenes) a los usuarios desde el servidor más cercano geográficamente para optimizar velocidades de carga.
ASGI (Asynchronous Server Gateway Interface)
Estándar de interfaz de servidor que permite la comunicación asíncrona en Python, permitiendo procesar múltiples conexiones simultáneas de forma ágil (usado por FastAPI/Uvicorn).
Multi-stage Build (Docker)
Estrategia de Docker que utiliza múltiples fases de construcción en un único Dockerfile para compilar binarios y copiar solo lo estrictamente necesario a la imagen de ejecución final, reduciendo su peso físico.
GitOps
Práctica operativa que utiliza repositorios de Git como la única fuente de verdad para definir y automatizar la infraestructura y los pipelines de despliegue continuo.
