Brechas Corregidas 19 Resueltas

El ecosistema transaccional MitrufelyWeb fue sometido a un análisis riguroso de seguridad combinando técnicas DAST (escaneo automatizado dinámico no autenticado) con auditorías de código estático (SAST) y manuales. Esta evaluación permitió detectar vulnerabilidades lógicas, fugas de secretos y configuraciones permisivas de red, las cuales fueron mitigadas para proteger los activos de la plataforma en producción.

1. Resumen Ejecutivo

El análisis arrojó inicialmente un total de 35 hallazgos de seguridad. Solo 4 fueron detectados por el escaneo automatizado de OWASP ZAP debido a las limitaciones de acceso en rutas protegidas. El resto se identificó mediante auditoría de código manual.

Tras la implementación de parches de seguridad, se lograron resolver **19 brechas de seguridad** críticas y altas, reduciendo la exposición al riesgo en aproximadamente un **49%**.

2. Metodología de las Pruebas

La evaluación se dividió en tres fases consecutivas:

  • Fase 1: Escaneo Dinámico Automatizado (DAST): OWASP ZAP 2.17.0 escaneó la superficie no autenticada de http://localhost:8000, descubriendo 131 endpoints. Debido a la protección de tokens, el 97% de las respuestas a peticiones de ZAP arrojaron código 4xx (acceso denegado).
  • Fase 2: Auditoría de Código Estática/Manual (SAST): Inspección minuciosa del repositorio (FastAPI en backend, Zustand e interfaces en frontend y archivos de infraestructura Docker/Nginx).
  • Fase 3: Implementación y Validación: Aplicación de correcciones de software siguiendo estándares OWASP ASVS y mitigaciones basadas en CWE.

3. Resultado del Escaneo ZAP — Antes vs Después

El escaneo original reportó 4 alertas. Tras la re-estructuración de la API, todas las alertas automáticas han sido resueltas en el código:

ID Alerta OWASP ZAP Severidad Estado Acción y Corrección Aplicada
1 Format String Error 🟡 Medio ✔ Resuelto Validación estricta de estructura JWT + paso mediante parámetros seguros en httpx.get.
2 Falta X-Content-Type-Options 🟢 Bajo ✔ Resuelto Inyección sistémica del header nosniff en todas las respuestas del backend.
3 Token de sesión en URL 🔵 Info ✔ Mitigado El token de verificación cuenta con un JTI de uso único guardado en Redis (no reutilizable).
4 Petición de Auth Identificada 🔵 Info ✔ Informativo Identificación correcta de endpoint por ZAP; no requiere acción técnica.

Análisis de Vulnerabilidad: Format String Error

Antes: El endpoint de Google Login interpolaba el id_token directamente en la petición HTTP externa sin validación. Al enviar secuencias maliciosas (ej. %n%s), el cliente HTTP de Python fallaba arrojando un error interno 500 no controlado.

Después: Se implementó un validador que comprueba que el token de Google cumpla la estructura estándar JWT (3 segmentos separados por puntos) antes de iniciar la solicitud, previniendo errores de formateo:

# Validación temprana de formato del token
parts = id_token.split(".")
if len(parts) != 3:
    raise InvalidTokenError("Token de Google con formato inválido")

# Envío seguro como parámetro codificado automáticamente
response = await client.get(url, params={"id_token": id_token})

4. Hallazgos de la Auditoría de Código

La auditoría manual identificó problemas adicionales de lógica de negocio, manejo de sesiones y secretos commiteados en el repositorio.

Hallazgos Críticos y Altos

ID Descripción del Riesgo Antes Después Mitigación Realizada
C-01 Auto-admin por dominio de email sin verificación 🔴 Vulnerable ✔ Corregido Eliminada lógica de auto-promoción; todos se registran con rol CLIENTE inactivo.
C-03 Tokens JWT expuestos en sessionStorage 🔴 Vulnerable ✔ Corregido Access token vive en memoria RAM de Axios; sessionStorage solo guarda estado UI.
C-04 Archivo .env expuesto en control de versiones 🔴 Vulnerable ✔ Corregido Removido del historial mediante git, añadido a gitignore y creado .env.example.
H-03 Ausencia de Rate Limiting global en API 🟠 Vulnerable ✔ Corregido Integración de slowapi con backend de Redis para bloqueo de abusos de IPs.
H-05 Enumeración de cuentas registradas en /register 🟠 Vulnerable ✔ Corregido Respuesta idéntica controlada (200 OK) sin revelar si el correo electrónico existía.
H-06 Cierre de sesión no invalidaba token en servidor 🟠 Vulnerable ✔ Corregido Creación de hook useLogout que llama a POST `/auth/logout` para registrar en Redis.

5. Detalle de Correcciones por Componente

A. Backend FastAPI (FastAPI)

  • SecurityHeadersMiddleware: Se inyectan de forma centralizada cabeceras anti-clickjacking y anti-XSS (X-Content-Type-Options: nosniff, X-Frame-Options: DENY, Content-Security-Policy) en cada respuesta HTTP del backend.
  • Rate Limiting (slowapi): Endpoints críticos limitados (Registro: 5/min, Google Login: 10/min, Refresh token: 30/min).
  • CORS Restringido: Eliminación del comodín ALLOWED_HEADERS=["*"] cuando las credenciales están habilitadas, forzando cabeceras explícitas de Authorization y Content-Type.

B. Frontend SPA (React)

  • Tokens en Memoria: El access token se guarda en una variable local de JS (memoria no persistente) y nunca se almacena en `sessionStorage` ni `localStorage` para evitar exfiltraciones en caso de ataques XSS.
  • Hook de Cierre de Sesión Unificado: Implementación de useLogout en 10 vistas principales que llama al servicio de invalidación de tokens del servidor en Redis antes de limpiar la sesión en el navegador.

C. Infraestructura y Docker

  • nginx.conf de Producción: Adición de CSP restringida, HSTS (Strict-Transport-Security) y desactivación de firmas del servidor nginx en las cabeceras HTTP.
  • Token de Webhook Delivery: Se removieron contraseñas débiles por defecto en main.py y se configuró un token asimétrico requerido en las variables de entorno de producción.

6. Mapa de Riesgo Residual Remanente

A pesar de la remediación del 49% de las brechas de seguridad, existen riesgos de infraestructura y configuraciones criptográficas que deben ser abordados en el corto y mediano plazo:

ID Severidad Descripción del Riesgo Residual Acción de Mitigación Requerida
C-02 🔴 Crítico Secretos de producción (Base de datos, claves JWT) expuestos en texto plano en el archivo .env en disco. Migrar la configuración de credenciales a un manejador de secretos externo (Vault o AWS Secrets Manager).
H-01 🟠 Alto El cambio de contraseña no invalida los tokens JWT existentes de otros dispositivos. Implementar el control de token_version en el modelo de base de datos de usuarios.
H-02 🟠 Alto Firma de JWT basada en clave simétrica compartida (HS256). Migrar el algoritmo de firma a criptografía asimétrica (RS256 o EdDSA).
M-11 🟡 Medio El contenedor de Celery se ejecuta como root (C_FORCE_ROOT). Reconfigurar el Dockerfile para crear y usar un usuario de sistema sin privilegios.
M-13 🟡 Medio Endpoints del microservicio de Delivery no requieren autenticación previa de red. Configurar políticas de firewall o API Gateways para restringir las llamadas a la subred interna.

7. Conclusiones y Guía de Verificación

Conclusión General: Las correcciones aplicadas redujeron la exposición del proyecto MitrufelyWeb de un estado de riesgo alto a un riesgo medio-bajo controlable. El escaneo dinámico automático (ZAP) fue útil para identificar errores de input, pero resultó insuficiente sin la auditoría manual de código, la cual expuso el 89% de las brechas de mayor severidad.

Guía para Re-verificar Parches de Seguridad

1. Verificación de Inyección de Cabeceras HTTP:

Puedes verificar que las respuestas contengan cabeceras anti-XSS y HSTS ejecutando una petición cURL al backend:

curl -I http://localhost:8000/api/v1/auth/login

Debe responder con cabeceras explícitas de: X-Content-Type-Options: nosniff y X-Frame-Options: DENY.

2. Ejecutar Pruebas Automatizadas de Seguridad:

Para comprobar que los parches de validación de tokens y JWT sigan funcionando correctamente sin regresiones:

cd _backEnd
.venv\Scripts\python -m pytest tests/unit/test_auth_service.py -v
📖 Glosario

📖 Glosario de Seguridad

DAST (Dynamic Application Security Testing)
Pruebas dinámicas realizadas desde el exterior de la aplicación en tiempo de ejecución (ej. escaneo con OWASP ZAP).
SAST (Static Application Security Testing)
Pruebas estáticas de caja blanca que analizan el código fuente sin ejecutar la aplicación para buscar fallos de sintaxis o patrones inseguros.
CWE (Common Weakness Enumeration)
Diccionario formal que define y cataloga debilidades de seguridad del software (ej. CWE-79 para XSS).
XSS (Cross-Site Scripting)
Vulnerabilidad que permite a un atacante inyectar scripts maliciosos en páginas web visitadas por otros usuarios, comúnmente con el fin de robar tokens de sesión de sessionStorage.
Rate Limiting
Mecanismo de control de tráfico que restringe la cantidad de solicitudes que un cliente puede hacer a la API en un período, mitigando ataques de denegación de servicio (DoS) y fuerza bruta.
JTI (JWT ID)
Identificador único asignado a un token JWT para prevenir ataques de replay, asegurando que un token sea de uso único.
